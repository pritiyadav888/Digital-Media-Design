Week 2: Building Blocks of the Web

Welcome to Week 2! This module dives deeper into the fundamental building blocks of web pages. You'll learn about the essential HTML elements used to structure content, how the internet and web truly function, key concepts that guide modern web design, and how to incorporate interactive forms and rich media into your sites.

By the end of this week, you'll have a solid understanding of not just *what* HTML tags do, but *why* they are used in specific ways to create accessible, functional, and well-performing web experiences.

HTML Document Structure: The Blueprint

Every webpage starts with a foundational HTML structure. Think of it as the architectural blueprint that tells the browser how to organize all the content.

HTML elements are the building blocks, identified by **tags** like `<p>` for a paragraph or `<h1>` for a main heading. Most elements come in pairs: an opening tag and a closing tag, wrapping around the content. 

<p>This is a paragraph.</p>
<h1>Main Title Here</h1>

The browser reads these tags, but only displays what's *between* them. A common beginner mistake is forgetting the closing slash (`/`) in a tag, which can make your page look very strange!

**Pro Tip:** While HTML isn't case-sensitive (`<P>` works just like `<p>`), always use lowercase for consistency and modern best practices.

Every HTML5 document should start with this basic, essential structure. It's the bare minimum for a well-formed webpage: 

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>My Awesome Page</title>
</head>
<body>
    <!-- All your visible content goes here -->
    <h1>Hello, Web!</h1>
    <p>This is my first structured page.</p>
</body>
</html>
  • `<!DOCTYPE html>`: The very first line, telling the browser this is an HTML5 document.
  • `<html lang="en">`: The root of your page, indicating the document's language (important for accessibility).
  • `<head>`: Contains invisible information about the page, like its title, character set (`UTF-8` for broad character support), and mobile responsiveness settings.
  • `<title>`: The text that appears in the browser tab or window title. Crucial for users and search engines!
  • `<body>`: This is where all the visible content of your webpage lives – headings, paragraphs, images, videos, everything users see.

**Mini-Activity:** Try changing the text inside the `<title>` tags in your own `index.html` file and see how your browser tab changes!

How you name your files matters! Good naming makes your project organized and prevents broken links.

  • **Proper endings (suffixes):** HTML files need `.html` or `.htm`. Images use `.png`, `.jpg`, `.gif`, or `.svg`. Videos use `.mp4`, `.webm`, etc.
  • **No spaces:** Use hyphens (`-`) or underscores (`_`) instead. `my-awesome-page.html` is good; `my awesome page.html` is bad.
  • **Avoid special characters:** Stick to letters, numbers, hyphens, and underscores.
  • **Lowercase is king:** Always use lowercase for filenames. It prevents issues on different web servers.
  • **Keep it short & descriptive:** `about-us.html` is better than `a-long-document-about-our-company.html`.

Even pros make mistakes! Here are common reasons your webpage might not look right, and how to debug them:

  • **"My changes aren't showing!"** Did you save the file? Did you refresh the browser? Is the file in the correct folder?
  • **"Half my page disappeared!"** You likely missed a closing tag (`</p>`) or a closing bracket (`>`) or a quote mark (`"`) in your HTML. The browser gets confused and stops rendering.
  • **"Broken image icon!"** The browser can't find your image. Double-check the image file name and its location. Is it in the same folder, or did you specify the correct subfolder?
  • **Browsers ignore extra whitespace:** Multiple spaces, line breaks, and tabs in your HTML source don't affect how the page looks. They're just for making your code readable.

**Your Best Friends for Debugging:**

  • **Browser Developer Tools (Console tab):** Press `F12` (Windows/Linux) or `Cmd+Option+I` (Mac) and look for red error messages.
  • **Online HTML Validators:** Tools like the W3C Markup Validation Service check your HTML against official rules and tell you exactly where your errors are.

Marking Up Text: Giving Content Meaning

HTML's main job is to add meaning (semantics) and structure to your content. It's *not* about making it look pretty (that's CSS's job!). Always choose the HTML element that best describes what your content *is*.

  • `<p>` (Paragraph): Your go-to for basic blocks of text. Browsers automatically put paragraphs on new lines with some space above and below.
  • `<h1>` to `<h6>` (Headings): Used for titles and subtitles. `<h1>` is for the most important heading on a page (usually only one per page!), and `<h6>` is the least important. Use them in a logical order (`h1` then `h2`, etc.) to create a clear outline for your page, which helps both users (especially with screen readers) and search engines.

**Golden Rule:** All visible text on your webpage *must* be inside an HTML element. No "naked text" allowed!

The `<hr>` element represents a "thematic break" – a significant shift in topic or section within a document. Historically, it displayed as a horizontal line, and many browsers still do. Semantically, it's about indicating a logical separation in content. 

<p>Here's some introductory text about our new product.</p>
<hr>
<p>Now, let's dive into the technical specifications.</p>

Humans love lists! HTML gives us three ways to structure them:

  • Unordered Lists (`<ul>`): For items where the order doesn't matter (like a grocery list). Each item is an `<li>` (list item). Browsers usually show these with bullet points.
  • Ordered Lists (`<ol>`): For items where the sequence is important (like step-by-step instructions). Each item is also an `<li>`. Browsers usually number these automatically.
  • Description Lists (`<dl>`): For pairs of names and values (like terms and their definitions). Uses `<dt>` (description term) for the name and `<dd>` (description details) for the value. 
<!-- Unordered List Example -->
<ul>
    <li>Apples</li>
    <li>Bananas</li>
</ul>

<!-- Ordered List Example -->
<ol>
    <li>Boil water</li>
    <li>Add pasta</li>
</ol>

<!-- Description List Example -->
<dl>
    <dt>HTML</dt>
    <dd>HyperText Markup Language</dd>
    <dt>CSS</dt>
    <dd>Cascading Style Sheets</dd>
</dl>

**Mini-Activity:** Think of 3 steps for making your favorite drink. How would you mark that up as an ordered list in HTML?

Beyond basic text, HTML5 introduced powerful **semantic elements** to give meaning to larger sections of your webpage. This helps browsers, search engines, and assistive technologies understand the purpose of different content blocks.

  • `<main>`: Identifies the primary, unique content of the page. There should only be one `<main>` per page.
  • `<header>`: Introductory content for a page, a section, or an article. Often contains logos, site titles, and main navigation.
  • `<nav>`: Defines a section containing navigation links (e.g., your main menu).
  • `<section>`: A thematic grouping of content. Think of it as a logical chapter or distinct segment within your page.
  • `<article>`: Encapsulates self-contained content that could be independently distributed or reused, like a blog post, news story, or user comment.
  • `<aside>`: Content that is tangentially related to the main content, often presented as a sidebar (e.g., related links, advertisements, pull quotes).
  • `<footer>`: Concluding content for a page, section, or article. Typically includes copyright info, contact details, or secondary navigation.
  • `<address>`: Provides contact information for the author/owner of the nearest `<article>` or `<body>` element.

**Mini-Activity:** Look at a favorite news website. Can you identify where they might use `<header>`, `<nav>`, `<article>`, and `<aside>`?

These elements apply semantic meaning or visual emphasis to phrases *within* a block of text. They "go with the flow" and don't start new lines.

  • `<em>` (Emphasis): For stressed emphasis (e.g., "This is *really* important!"). Often italicized by default.
  • `<strong>` (Strong Importance): For content of strong importance, seriousness, or urgency (e.g., "**Warning:** Do not proceed!"). Often bold by default.
  • `<b>` (Bold): For visually emphasized text without strong semantic importance (e.g., keywords in a review).
  • `<i>` (Italic): For alternative voice, technical terms, or foreign words.
  • `<u>` (Underline): For formally named entities or misspelled words (avoid for links!).
  • `<s>` (Strikethrough): For text that is no longer accurate or relevant.
  • `<small>`: For fine print, disclaimers, or copyright notices.
  • `<code>`: For displaying computer code snippets.
  • `<var>`: For variables in mathematical expressions or programming.
  • `<samp>`: For sample program output.
  • `<kbd>`: For user-entered keyboard input.
  • `<sub>` (Subscript) / `<sup>` (Superscript): For chemical formulas (`H₂O`) or mathematical equations (`E=MC²`).
  • `<mark>`: For text that is contextually relevant, like highlighting search terms.
  • `<time>`: For machine-readable dates and times (e.g., `<time datetime="2026-01-15">January 15th, 2026</time>`).
  • `<br>` (Line Break): Inserts a single line break. Useful for addresses or poetry.
  • `<wbr>` (Word Break): Suggests a safe place for a browser to break a very long word if it needs to wrap.

When no other semantic HTML element perfectly describes your content, `<div>` and `<span>` come to the rescue. They are "generic" containers that get their meaning from `id` and `class` attributes.

  • `<div>` (Division): A generic **block-level** container. It's often used to group larger sections of content for layout purposes or to apply styles to a whole block. Think of it as a flexible box that starts on a new line.
  • `<span>` (Span): A generic **inline** container. It's used to group smaller pieces of text or other inline elements, often to apply a specific style to just a word or phrase within a line. It doesn't start a new line. 
<div id="product-card">
    <h3>Product Name</h3>
    <p>Price: <span class="price-highlight">$29.99</span></p>
</div>

These attributes are essential for giving your HTML elements unique identities or grouping them for styling and JavaScript interactions.

  • `id` attribute: Assigns a **unique identifier** to a single element on the page. Like a person's unique ID number. You should only use an `id` value once per HTML document. Great for direct targeting by JavaScript or for creating links to specific sections (`<a href="#section-id">`).
  • `class` attribute: Classifies elements into **conceptual groups**. Like assigning students to different "classes" (e.g., "math-class", "science-class"). Multiple elements can share the same class. This is perfect for applying the same styles to many elements (e.g., all buttons look the same) or for JavaScript to select multiple related elements.

Both `id` and `class` are **global attributes**, meaning you can use them on almost any HTML element.

ARIA (Accessible Rich Internet Applications) is a set of special attributes (`role`, `aria-label`, `aria-hidden`, etc.) that you can add to HTML elements. Their purpose is to provide extra semantic information for **assistive technologies** like screen readers, especially for dynamic content or custom UI components that might not have built-in accessibility. 

<div role="alert">Your message has been sent successfully!</div>
<button role="switch" aria-checked="false">Dark Mode Toggle</button>

ARIA helps describe the purpose and state of elements to users who can't see or interact with them visually, ensuring a more inclusive web experience.

Sometimes, you need to display characters that HTML might confuse with its own code (like `<` or `>`). To do this, you use **character entity references** (also called character escapes).

  • They start with an ampersand (`&`) and end with a semicolon (`;`).
  • **Named entities:** Use an easy-to-remember name (e.g., `<` for `<`).
  • **Numeric entities:** Use a number code (e.g., `<` for `<`). 
<p>To display the less-than symbol, use &lt;</p>
<p>Copyright &copy; 2026</p>

This ensures your browser displays the character correctly instead of trying to interpret it as part of an HTML tag.

How the Web Works: The Internet's Magic

Have you ever wondered what happens when you type a website address and press Enter? It's a fascinating journey involving the internet and the web. While often used interchangeably, they're distinct!

  • The Internet: This is the massive, global network of connected computers. It's the physical infrastructure (cables, routers, servers) and the rules (protocols) that allow any computer to talk to any other computer. Think of it as the entire road system.
  • The Web (World Wide Web): This is just *one* of the many ways information is shared *over* the internet. It's unique because it uses **hypertext links** to connect documents (webpages) to each other, forming a "web" of information. It uses a specific set of rules called **HTTP** (HyperText Transfer Protocol). Think of it as the specific rules for driving on those roads to visit websites.

The Web was invented in 1989 by Tim Berners-Lee at CERN, a particle physics lab. Its ongoing development is guided by the W3C (World Wide Web Consortium).

Every time you visit a website, your browser and a server perform a little dance:

  • The Server: A powerful computer (or specialized software) that "serves up" webpages and files when requested. It's always waiting for your requests.
  • Your Browser (the Client): The software you use (like Chrome, Firefox, Safari) that "requests" webpages from servers and then displays them for you.

**The Process:**

  1. You type a URL or click a link.
  2. Your browser sends an **HTTP Request** to the server.
  3. The server finds the requested files (HTML, CSS, images, etc.) and sends them back as an **HTTP Response**.
  4. Your browser's **rendering engine** takes all these pieces and assembles them into the beautiful webpage you see.

**IP Addresses & DNS (The Phonebook of the Internet):** Every device connected to the internet has a unique numeric **IP address** (e.g., `192.168.1.1`). Since these are hard to remember, the **DNS (Domain Name System)** acts like a phonebook, translating easy-to-remember **domain names** (like `google.com`) into their corresponding IP addresses.

A **URL (Uniform Resource Locator)** is the unique address for every single webpage or resource on the internet. It's how your browser knows exactly where to find something.

**Anatomy of a Modern URL:**

https://www.example.com/blog/2026/my-first-post.html
|         |             |        |           |
Protocol  Domain Name   Path     Filename
  • **Protocol (`https://`):** The rules for communication. `HTTPS` (HyperText Transfer Protocol Secure) is the standard today, encrypting data for security. Always use `HTTPS` for your sites!
  • **Domain Name (`www.example.com`):** The human-readable name of the website.
  • **Path (`/blog/2026/`):** The folder structure on the server where the file is located.
  • **Filename (`my-first-post.html`):** The specific file being requested.

**Did you know?** If a URL ends with a folder name (e.g., `example.com/blog/`), the server often looks for a default file inside that folder, usually named `index.html`.

What you see as one seamless webpage is actually an assembly of multiple files by your browser:

  • **HTML Document:** The main blueprint. It contains the text content and tells the browser where to find all the other pieces.
  • **Style Sheets (CSS):** Separate files that define how all your HTML elements should look (colors, fonts, layout).
  • **JavaScript Files:** Separate files that add interactivity and dynamic behavior.
  • **Images/Videos/Audio:** Individual media files linked within the HTML.

Your browser's **rendering engine** is the magic worker that gathers all these separate files and pieces them together into the final, interactive webpage you experience. This usually happens in milliseconds!

Big Concepts in Web Design: Navigating the Modern Web

As a web designer in 2026, you're building for a diverse and ever-changing digital landscape. These "Big Concepts" are guiding principles to help you create effective, robust, and user-friendly websites.

Gone are the days when everyone viewed websites on a desktop computer. Today, users access the web from countless devices: smartphones, tablets, smartwatches, smart TVs, gaming consoles, and even car dashboards!

**Key Takeaway:** You can *never* assume how your design will be viewed. Always design for flexibility and adaptability across all possible screen sizes, input methods (touch, mouse, voice), and network speeds. **Mobile-first** thinking (designing for small screens first, then scaling up) is a common and effective strategy.

Web standards are like the agreed-upon rules for building websites, documented by the W3C (World Wide Web Consortium). Sticking to these standards is absolutely crucial for:

  • **Consistency:** Your site will look and behave predictably across different browsers (Chrome, Firefox, Safari, Edge, etc.).
  • **Future-Proofing:** Standards help ensure your content remains readable and functional as web technologies evolve.
  • **Accessibility:** Standard-compliant code is easier for assistive technologies to understand.

Building with standards reduces headaches and creates a more reliable web experience for everyone.

This strategy means you start by building a webpage that provides a solid, functional "baseline experience" for *all* users, even those with older browsers or slower connections. Then, you progressively add more advanced features, visual flair, and interactivity for users whose browsers can support them.

**Think of it like this:** Everyone gets the basic information (the core content), but those with modern devices get extra "bells and whistles" (animations, interactive maps, etc.). This ensures your site is accessible and usable by the widest possible audience.

RWD is the primary strategy for making your website look great on any screen size. Instead of creating separate websites for desktop and mobile, you design one website that **responds** to the size of the user's browser window (the "viewport").

This is achieved using CSS **media queries** (which apply different styles based on screen width), flexible images, and fluid layouts (using percentages or modern CSS Grid/Flexbox). Your content might rearrange, images might resize, and navigation might change from a menu bar to a "hamburger" icon, all on the same single HTML page.

Accessibility means designing and building webpages that are usable by *everyone*, regardless of their abilities or disabilities. This includes users with visual impairments (who might use screen readers), mobility impairments (who might use keyboards or voice commands), auditory impairments, or cognitive impairments.

**Why it matters:** It's not just about compliance; it's about inclusivity. Accessible sites reach a wider audience and often perform better in search engines. The W3C's Web Accessibility Initiative (WAI) provides guidelines (WCAG) to help developers achieve this.

Site performance is simply how fast your webpage loads and becomes interactive for the user. In today's fast-paced world, speed is critical:

  • Users expect sites to load in under 2 seconds.
  • Slow sites lead to users leaving and not returning.
  • Search engines (like Google) prioritize faster sites in their rankings.

**Tips for Speed:** Optimize images (make them smaller), minimize HTML/CSS/JS code, reduce unnecessary "trips" to the server (HTTP requests), and load scripts efficiently.

HTML Tables: Organizing Data

HTML tables are specifically designed for presenting **tabular data** – information arranged neatly in rows and columns. Think schedules, product comparisons, or financial figures. **Important:** Tables are for data, *not* for arranging your entire webpage layout (we use CSS Grid or Flexbox for that now!).

A simple table is built with these core elements:

  • `<table>`: The main container for your entire table.
  • `<tr>` (Table Row): Defines a single row in your table.
  • `<th>` (Table Header): Defines a header cell. These usually appear bold and centered by default, providing labels for columns or rows.
  • `<td>` (Table Data): Defines a standard data cell. This is where your actual information goes. 
<table>
    <tr>
        <th>Item</th>
        <th>Quantity</th>
    </tr>
    <tr>
        <td>Apples</td>
        <td>5</td>
    </tr>
</table>

The number of columns in your table is automatically determined by how many `<th>` or `<td>` cells you put in each row.

Sometimes, a single cell needs to stretch across multiple columns or rows. This is where `colspan` and `rowspan` attributes come in handy:

  • `colspan="number"`: Makes a cell span horizontally across the specified number of columns.
  • `rowspan="number"`: Makes a cell span vertically down the specified number of rows. 
<table>
    <tr>
        <th colspan="2">Product Details</th> <!-- This header spans 2 columns -->
    </tr>
    <tr>
        <td>Name</td>
        <td>Widget X</td>
    </tr>
    <tr>
        <td rowspan="2">Specs</td> <!-- This cell spans 2 rows -->
        <td>Weight: 1kg</td>
    </tr>
    <tr>
        <td>Color: Blue</td>
    </tr>
</table>

Tables can be tricky for users with screen readers (software that reads webpage content aloud). To make your tables accessible:

  • `<caption>`: Always provide a descriptive title or summary for your table using the `<caption>` element, placed immediately after the `<table>` tag.
  • `scope` attribute on `<th>`: Explicitly tell the browser if a header (`<th>`) applies to a `row` or a `col` (column). This helps screen readers correctly associate data with its headers. 
<table>
    <caption>Monthly Sales Performance (Q1 2026)</caption>
    <tr>
        <th scope="col">Month</th>
        <th scope="col">Revenue</th>
    </tr>
    <tr>
        <th scope="row">January</th>
        <td>$50,000</td>
    </tr>
</table>

**Mini-Activity:** Imagine a simple table for a class schedule (e.g., "Time" and "Subject"). How would you use `<caption>` and `scope` attributes?

For more complex tables, you can semantically group rows and columns to provide even more structure and make them easier to style or interact with using JavaScript:

  • `<thead>`: Groups the header rows of a table.
  • `<tbody>`: Groups the main body rows of a table (you can have multiple `<tbody>` elements).
  • `<tfoot>`: Groups the footer rows of a table (e.g., totals, footnotes).
  • `<colgroup>`: Groups columns together for styling or semantic purposes.
  • `<col>`: Defines properties for one or more columns within a `<colgroup>`. 
<table>
    <thead>
        <tr><th>Product</th><th>Price</th></tr>
    </thead>
    <tbody>
        <tr><td>Laptop</td><td>$1200</td></tr>
    </tbody>
    <tfoot>
        <tr><td colspan="2">All prices in USD</td></tr>
    </tfoot>
</table>

Web Forms: Collecting User Input

Web forms are the backbone of interaction on the internet, allowing users to submit information, make purchases, sign up for services, or search for content. They combine visible HTML controls with behind-the-scenes server-side processing.

When you fill out and submit a web form, here's the simplified journey your data takes:

  1. You interact with HTML form controls (like text boxes, buttons, checkboxes) on a webpage.
  2. You click a "Submit" button.
  3. Your browser gathers all the information you entered, encodes it, and sends it as an **HTTP Request** to a specific server-side application (a program running on the web server).
  4. The server-side application processes your data (e.g., saves it to a database, sends an email, performs a calculation).
  5. The server sends an **HTTP Response** back to your browser, often a "Thank You" page or updated content.

**Mini-Activity:** Think about the last time you signed up for an online service. What kind of information did the form ask for? What happened after you clicked "Submit"?

The `<form>` element is the essential container for all your form controls and content. All inputs, buttons, and text related to a single submission should be inside it. You cannot nest one `<form>` inside another.

  • `action` attribute: This tells the browser *where* to send the form data (the URL of the server-side script).
  • `method` attribute: This defines *how* the data is sent.
    • `GET`: Data is appended to the URL (visible in the browser's address bar). Best for simple searches or data that isn't sensitive. Data size is limited.
    • `POST`: Data is sent in the body of the HTTP request (not visible in the URL). Best for sensitive information (passwords, personal details) and large amounts of data (like file uploads). This is the most common method for forms. 
<form action="/submit-contact-form.php" method="POST">
    <!-- Your form inputs and buttons go here -->
</form>

These are the building blocks users interact with:

  • `<input type="text">` (Single-line Text Field): For short text input.
    • `name`: (Crucial!) The name of the data piece (e.g., `username`).
    • `value`: Default text that appears in the field when the page loads.
    • `placeholder`: Light gray hint text that disappears when the user types (e.g., "Enter your name").
    • `maxlength`, `minlength`: Set character limits.
  • `<textarea>` (Multi-line Text Area): For longer text input (comments, messages).
    • `rows`, `cols`: Define the visible height (in lines) and width (in characters).
  • `disabled` attribute: Makes a control unusable and grayed out. Its value is *not* submitted with the form.
  • `readonly` attribute: Makes a control uneditable by the user, but its value *is* submitted with the form. 
<input type="text" name="fullName" placeholder="Your Full Name" maxlength="50">
<textarea name="feedback" rows="5" cols="50" placeholder="Your comments..."></textarea>
<input type="text" value="Default Text" readonly>

Embedded Media: Beyond Static Images

HTML allows you to embed rich media and external resources directly into your webpages, bringing dynamic content like videos, audio, and interactive maps to your users.

The `<iframe>` (inline frame) element allows you to embed another HTML document or web resource (like a YouTube video, Google Map, or even an ad) directly within your current page. It creates a "window within a window." 

<!-- Example: Embedding a YouTube video -->
<iframe width="560" height="315" src="https://www.youtube.com/embed/dQw4w9WgXcQ" 
    frameborder="0" allow="accelerometer; autoplay; clipboard-write; encrypted-media; gyroscope; picture-in-picture" 
    allowfullscreen></iframe>

<!-- Example: Embedding a Google Map -->
<iframe src="https://www.google.com/maps/embed?pb=!1m18!1m12!1m3!1d3153.254391696239!2d-122.4194156846813!3d37.77492977975924!2m3!1f0!2f0!3f0!3m2!1i1024!2i768!4f13.1!3m3!1m2!1s0x8085808e8d8d8d8d%3A0x8085808e8d8d8d8d!2sGolden+Gate+Bridge!5e0!3m2!1sen!2sus!4v1678901234567!5m2!1sen!2sus"
    width="600" height="450" style="border:0;" allowfullscreen="" loading="lazy" referrerpolicy="no-referrer-when-downgrade"></iframe>

The `src` attribute specifies the URL of the content to embed. Attributes like `width`, `height`, and `frameborder` control its appearance. The `sandbox` attribute is important for security, restricting what the embedded content can do.

The `<object>` element is a generic way to embed external resources. Historically, it was used for content requiring browser plugins like Flash or Java Applets. While it's still part of HTML, its use has significantly decreased as modern browsers support media natively. 

<!-- Example (less common today, but shows usage) -->
<object data="your-flash-animation.swf" type="application/x-shockwave-flash" width="300" height="200">
    <!-- Fallback content if plugin not available -->
    <p>Your browser does not support Flash.</p>
</object>

For most modern media (video, audio, SVG), native HTML5 elements or `<iframe>` are preferred over `<object>`.

HTML5 introduced dedicated elements for embedding video and audio directly, without needing third-party plugins. This is the standard way to include multimedia today.

  • `<video>` element: For embedding video. 
    <video controls width="640" height="360" poster="video-thumbnail.jpg">
    <source src="my-video.mp4" type="video/mp4">
    <source src="my-video.webm" type="video/webm">
    <p>Your browser does not support the video tag.</p>
</video>
    • `controls`: Displays playback controls (play/pause, volume, fullscreen).
    • `autoplay`: Starts playing automatically (use *very* sparingly, it can be annoying!).
    • `loop`: Repeats the video.
    • `muted`: Disables audio by default.
    • `poster`: An image to display before the video plays.
    • `<source>`: Provides multiple video formats (`.mp4` with H.264, `.webm` with VP8/VP9) for wider browser compatibility.
  • `<audio>` element: For embedding audio. 
    <audio controls>
    <source src="my-audio.mp3" type="audio/mpeg">
    <source src="my-audio.ogg" type="audio/ogg">
    <p>Your browser does not support the audio tag.</p>
</audio>
    • `controls`: Displays playback controls.
    • Similar `autoplay`, `loop`, `muted` attributes apply.
    • `<source>`: Provides multiple audio formats (`.mp3`, `.ogg`, `.wav`) for compatibility.

The `<canvas>` element creates a blank, rectangular area on your webpage where you can draw graphics, animations, or even simple games entirely using **JavaScript**. It's a pixel-based drawing surface. 

<canvas id="myDrawingCanvas" width="300" height="150" style="border:2px solid #3e5c76; background-color: #f0ebd8;">
    Your browser does not support the canvas tag.
</canvas>
<script>
    // This JavaScript draws a simple circle on the canvas
    const canvas = document.getElementById('myDrawingCanvas');
    const ctx = canvas.getContext('2d'); // Get the 2D drawing context

    ctx.beginPath(); // Start drawing path
    ctx.arc(150, 75, 50, 0, 2 * Math.PI); // Draw a circle: (x, y, radius, startAngle, endAngle)
    ctx.fillStyle = '#748cab'; // Set fill color
    ctx.fill(); // Fill the circle

    ctx.lineWidth = 3; // Set line width
    ctx.strokeStyle = '#1d2d44'; // Set stroke color
    ctx.stroke(); // Draw the circle's outline
</script>

All drawing operations on a `<canvas>` are performed using JavaScript commands, making it incredibly powerful for creating dynamic and interactive visual experiences directly in the browser.
